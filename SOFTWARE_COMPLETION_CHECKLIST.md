## 0. 审阅边界与结论

### 0.1 本清单不做什么

- 不以继续增加分析算法或图表数量为目标。
- 不为了“显得完善”而重写已经稳定的 GUI、录音链或工程格式。
- 不把 Windows ARM64、macOS Apple Silicon、PhonRec 独立模式等已经确认的产品边界重新当成待讨论架构。
- 不要求为 macOS 打包 CLI；当前“CLI 仅随 Windows 套件发布”可以作为明确的产品边界保留。
- 不要求支持 Intel Mac；当前 macOS ARM64 / Apple Silicon 定位可以作为明确的产品边界保留。
- 不建议为了代码行数而立刻大规模拆分 `cli.py`、`toolkit.py`、`modules/app.py` 或 PhonRec 主文件。完工阶段优先冻结合同、补验收和补文档，避免收尾期引入大面积回归。
- 按项目要求，本清单不要求维护独立版本日志。版本说明直接放在 GitHub Release 页面与发布验收记录中即可。

### 0.2 当前已经具备的“成品骨架”

- [x] 主程序已经覆盖导入、声学分析、人工复核、异常提示、工程保存恢复与多种导出。
- [x] Toolkit 已形成独立工作台，覆盖音频合并/拆分、字表编辑、TextGrid 转换、工程预览、报告导出和受控脚本。

### 0.3 总体判断

功能层面已经接近完成。剩余工作主要不是“再做一个功能”，而是补齐正式软件的最后一公里：

1. 用户能否放心安装：授权、签名、公证、隐私与安全说明。
2. 用户能否不看源码也正确使用：首次使用、模式差异、数据位置、恢复方式、支持入口。
3. 发布能否稳定重复：版本单一事实源、依赖锁定、CI 门禁、发布说明和资产验收。
4. 数据能否长期保存：工程格式规范、兼容策略、真实旧工程夹具、恢复演练。
5. 出问题后能否定位：持久化日志、诊断包、问题模板、支持边界。

如果只做最关键的收尾，建议先完成本文所有 **P0** 项；完成 P0 后即可把项目视为“正式可交付的软件”，P1 用于把维护成本降下来，P2 则是后续体验增强。

## 1. 优先级定义

| 优先级 | 含义 | 建议时点 |
| --- | --- | --- |
| **P0** | 正式对外发布前应完成；关系到授权、信任、数据安全、安装成功或支持闭环 | 下一次正式发布前 |
| **P1** | 不阻止当前版本使用，但明显影响可维护性、可重复发布和专业完成度 | 下一至两个小版本内 |
| **P2** | 体验增强或扩大用户群，不应阻塞项目完工 | 稳定维护期按需求选择 |

  - 完成判据：仓库根目录存在明确授权文件；安装包或“关于”窗口可访问；PhonRec、Toolkit、CLI 与主程序的适用范围清楚。

  - 完成判据：形成 `THIRD_PARTY_NOTICES.md` 或随安装包分发的同等文档；每项包含名称、版本/来源、许可证和是否随包分发。
