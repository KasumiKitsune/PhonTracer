

# PhonTracer 使用手册

欢迎使用 PhonTracer！这不仅仅是一个“画一条音高曲线”的小工具，而是一个专门为语音学实验设计的全流程处理工作站。它的核心理念是将“录音、字表匹配、音节切分、基频数据采样、人工可视化修正、最终数据导出”这一整条枯燥的数据处理流水线，变得直观、可控且可复查。

无论你是习惯在图形界面中精雕细琢每一条声调曲线，还是希望借助代码和 AI 助手进行大批量的自动化提取，PhonTracer 都能为你提供得心应手的解决方案。

## 1. 核心概念与术语速查（读前必看）

在正式开始之前，我们先来理清几个在软件中频繁出现的概念。了解了它们，你会发现 PhonTracer 的运行逻辑其实非常符合人类的直觉。

> 💡 **术语小词典：**
> 
> * **基频（Fundamental Frequency, 简称 F0）：** 物理声学中反映声音高低的指标，单位是赫兹（Hz）。我们在语音学中常说的“声调的起伏”，在声学层面主要就是通过提取 F0 曲线来体现的。
> * **VAD（Voice Activity Detection，语音活动检测）：** 一种计算机算法。简单来说，就是让软件能够自动判断“录音里的这一秒钟是有人在说话，还是纯粹的环境底噪”。PhonTracer 处理长录音的第一步，就是靠它把大段录音“切萝卜”一样切成一个个带语音的片段。
> * **发音人（Speaker）：** 在实验中录音的具体对象。软件支持同时管理多个发音人，他们就像平行的工作区，各自的音频、修改进度和参数都不会互相干扰。
> * **词项（Item）与音节（Syllable）：** 软件处理数据的基本层级。一个“词项”对应你字表里的一行（比如一个词、一个短语或一句话）。软件拿到“词项”后，会根据规则把它进一步拆分成“音节”，并在每个“音节”的内部采集若干个 F0 数据点（比如默认采 11 个点）。
> * **TextGrid 格式：** 语音学界最常用的分析软件 Praat 所采用的标注文件格式。它里面记录了录音中哪一秒到哪一秒对应着什么字词。PhonTracer 支持无缝导入和导出这种格式。
> * **T 值（T-value）：** 一种用于消除发音人个体差异的数据归一化方法。举个例子，男低音和女高音读同一个调，绝对赫兹值天差地别。T 值通过对数转换，把所有人的音高都压缩映射到 0 到 5 的范围内（5 代表该发音人的最高音，0 代表最低音），这样不同人的声调走势就能放在同一张图里直接对比了。

## 2. 快速起步：选对入口，跑通流程

PhonTracer 并不是只有一个干巴巴的窗口，它其实包含了三个不同的入口，分别对应你在不同阶段的需求：

1. **软件主体（PhonTracer）：** 最核心的图形界面。适合绝大多数用户进行导入、参数调整、声谱图精修和数据图表导出。
2. **音频工具箱（AudioToolkit）：** 一个独立的音频前处理套件。如果你的录音材料太长需要拆分，或者太碎需要合并，在进主体软件之前先用它处理会事半功倍。
3. **命令行界面（PhonTracerCLI）：** 专为批处理、写脚本或是让 AI 编程工具（如 Codex、Trae）代劳而设计的纯文本交互界面。

### 2.1 软件启动方式

如果你有代码基础，直接从源码启动是最方便的。在项目目录中安装好 Python 依赖后，可以在终端输入以下命令：

* 启动软件主体：`python main.py`
* 启动音频工具箱：`python audio_toolkit.py`
* 启动命令行界面：`python cli.py`

如果你下载的是打包好的 Windows 或 macOS 版本，直接双击对应的 `.exe` 或 `.app` 程序图标即可。

### 2.2 最小可行示例：一分钟跑通流程

如果你不想看长篇大论，只想知道这软件到底怎么跑起来，请跟着做一遍这个极其简单的流程：

1. 启动软件主体（运行 `python main.py`）。
2. 在左侧面板选择 **长音频模式**。
3. 把你自己的一段包含多词语朗读的录音（比如 `我的录音文件.wav`）拖入软件窗口导入。
4. 接着导入与之对应的纯文本字表（比如 `我的字表.txt`）。
5. 在右侧的树状列表中看看有没有标记为“需检查”的词项。如果有，点击它，在中间的声谱图中确认一下红色的起止边界对不对。
6. 确认没问题后，点击右下角的导出，选择 `xlsx` 表格。
7. 点击“导出项目（.teproj）”，把当前进度保存下来。

只要这个流程跑通了，就说明软件在你的电脑上运行完美，接下来就可以放心地处理海量数据了。

## 3. 软件主界面漫游指南

第一次打开 PhonTracer，你可能会被满屏的按钮和图表唬住。别担心，界面的布局逻辑极其清晰，它从左到右分为三个大区：“指挥中心”、“精修车间”和“成果仓库”。

### 3.1 左侧面板：全局指挥中心

这里掌管着软件的宏观运行状态。

* **发音人管理：** 你可以添加发音人、切换当前发音人，或者开启“统一字表”功能（多发音人读同一份材料时，开启它能省去重复导入字表的麻烦）。
* **模式与引擎切换：** 决定当前是处理一个长音频还是批量小音频；决定使用经典的 Praat 还是 REAPER 来作为底层测音高的引擎。
* **全局参数调整：** 所有的算法核心参数（如基频上下限、采样点数等）都在这里设定。改动后，可以点击“重新自动识别”让全项目应用新参数。

### 3.2 中间面板：可视化精修车间

当你需要像拿着放大镜一样检查某个词的具体表现时，中间面板就是你的主战场。
当你选中某个词项后，这里会显示该词的**声谱图**和**F0 音高曲线**。

* **红线与蓝线：** 红线代表这个词的起点和终点，蓝色的虚线代表词内部音节的交界线。发现切得不准？直接用鼠标拖动线条就可以手动校正，非常直观。
* **橡皮擦工具：** 这是实验中最常用的功能之一。有时候环境里突然有声噪音，导致原本平滑的音高曲线上飞出去几个异常的点。点击界面上的“橡皮擦”图标，鼠标会变成十字，在错误的 F0 点上轻轻一抹，这些数据就会被置为 0，不再参与后续的均值计算。
* **试听播放：** 提供了一个绿色的播放游标，你可以随时点击播放按钮，听一听当前的切分范围到底框住了什么声音。

### 3.3 右侧面板：项目树与成果仓库

右侧面板主要用于数据的组织、排查和最终的导出。

* **分组与词项管理：** 这里以树状图的形式展示了你所有的字词。你可以双击重命名，或者拖动词项把它们归类到不同的分组里。
* **智能筛查机制：** 这个功能非常关键！你可以在上方选择只筛选“需检查”或“已修改”的词项。“需检查”并不是一个真实的分组，而是软件提供的一种智能视图。如果某个词提取出的数据里存在大量的 0（没提取到音高），或者有数据计算异常（NaN），软件就会自动把它塞进“需检查”列表里，提醒你重点关照。
* **一键导出：** 所有修好的数据，都在这里的导出菜单生成最终结果。

## 4. 两大核心工作流与发音人管理

我们的录音材料千奇百怪，PhonTracer 为此设计了两套截然不同的工作流。但在介绍工作流之前，我们必须先弄懂“字表”是怎么一回事，因为它是贯穿一切的灵魂。

### 4.1 字表格式与音节拆分规则

字表可以是纯文本（`.txt`），也可以是从 TextGrid 里的 `words` 层转换过来的。最简单的字表就是一行写一个词。

**【示例 1】最简单的字表：**

```text
北京
上海
广州
```

**【示例 2】带分组标题的字表（方便后续按组统计）：**

```text
【普通话】
北京
上海

【英语】
bro/ther
sys/tem
```

*注：支持的分组标题形式包括 `【分组名】`、`[分组名]`、`# 分组名` 等。*

**【示例 3】一行写多个词（节约空间）：**

```text
北京 上海,广州，天津、南京
```

*注：你可以用空格、中英文逗号甚至顿号在同一行里隔开多个词。*

**音节拆分的玄机：**
软件拿到你的词后，是怎么知道它有几个音节的？它遵循以下智能规则：

1. **强行拆分符：** 如果你在词里写了斜杠 `/`，软件会无条件按照斜杠拆分。
2. **中文汉字：** 如果是纯中文，软件会自动把每个汉字当成一个音节拆开。
3. **外语单词：** 如果是连续的拉丁字母且没有斜杠，软件会把它当成一个整体（单音节）。

来看看直观的拆分效果表：

| 你在字表中写的文本    | 软件拆分出来的结果           |
|:------------ |:------------------- |
| `北京`         | `北`、`京`             |
| `bro/ther`   | `bro`、`ther`        |
| `computer`   | `computer` （作为一个整体） |
| `com/pu/ter` | `com`、`pu`、`ter`    |
| `天安/门`       | `天安`、`门`            |

> ⚠️ **注意：** 强烈建议你的字表里只写最纯净的文本标签，不要画蛇添足地写成 `001_北京`。因为如果你用了斜杠拆分（比如 `001_北京/大学`），软件会死板地把 `001_北京` 当成第一个音节，这显然不是你想要的。

### 4.2 长音频工作流（一段录音，包罗万象）

如果你让发音人坐在录音室里，一口气读完了几十上百个词，词与词之间有停顿，那这就是典型的长音频材料。

**操作步骤：**

1. 选择“长音频模式”，导入你的长录音文件（如 `.wav` 或 `.mp3`）。
2. 导入与录音顺序完全一致的字表。
3. 软件会开始狂奔：首先进行全段 VAD 语音活动检测，把一整段声音切成几十个小块；接着把字表里的词按顺序一个个扣在这些小块上；然后钻进每个词内部，寻找最响亮的元音区域；再切分内部音节；最后在每个音节里采点。

**可视化切分器：**
长音频由于容易受到咳嗽声、翻书声的干扰，自动切分可能会出现“多切了”或“少切了”的情况（比如检测出的语音段少于字表里的词数，软件会标注 `(缺失)`）。
此时你需要打开“可视化切分器”：

* **删除误检：** 对着被识别成语音的杂音段点右键，把它删掉。
* **手动框选：** 在空白的波形处点右键，可以手动新增一个片段。
* **调整边界：** 拖动红色竖线，确保片段正好包住语音。调整完确认后，软件会重新把字表按顺序对齐。

长音频模式还支持导入预先人工标注好的 **TextGrid**。如果你的 TextGrid 里有 `words` 层（词项边界）、`chars` 层（音节边界）甚至 `groups` 层，软件会优先提取这些极其精确的人工边界，完全跳过容易出错的自动 VAD 检测。

### 4.3 批量音频工作流（文件碎片化，化零为整）

如果你的录音已经被剪辑成了无数个独立的小文件（比如每个词对应一个音频），就需要用到批量模式。

**操作步骤：**
导入一批音频后，你可以直接使用，软件会把**文件名**当成词的名称，并提取整段音频的基频。但这种做法往往不够严谨，我们推荐在导入音频后，再导入一份字表，让软件将音频和字表“联姻”。

批量模式下导入字表有两种匹配逻辑，请根据你的情况对号入座：

| 匹配模式     | 适用场景                                           | 软件的处理行为                                              |
|:-------- |:---------------------------------------------- |:---------------------------------------------------- |
| **模糊匹配** | 文件名包含词项名称，但夹杂着各种编号或没用的后缀（比如 `001-北京(重录).wav`）。 | 软件会像侦探一样，忽略乱七八糟的字符，去寻找名字里包含“北京”的文件并对应起来。中文比汉字，英文比字母。 |
| **顺序匹配** | 文件名完全没有规律，但你在导入时文件的排序已经跟字表顺序一模一样了。             | 软件不再看名字，直接按排队的顺序，第一个音频对应字表第一个词，严丝合缝地扣上去。             |

在批量模式下，同样支持批量导入 TextGrid。软件会优先尝试找同名的 TextGrid，如果找不到，就按顺序兜底。

### 4.4 发音人管理与统一字表

研究往往需要采集多个发音人的数据。左侧的发音人列表就是你的“分身术”。
你可以不断添加新的发音人（比如“发音人A”、“发音人B”）。切换发音人时，那一瞬间，中间的声谱图、右侧的项目树、该发音人修改过的参数，都会像魔法一样恢复到他专属的状态。

如果你在做同一个实验，几十个发音人读的都是同一份字表，请务必在左侧开启 **“统一字表”**。这样你只要为第一个发音人导入过字表，后续再导入新发音人的音频时，软件就能自动套用字表和匹配规则，大幅提升效率。

## 5. 算法揭秘与参数调优（让你真正懂它）

要让提取出的数据精准可靠，不能只靠默认设置，你必须了解左侧那一排参数究竟在控制什么。让我们从算法原理出发，逐一击破。

### 5.1 宏观与微观切分的博弈

软件在找声音时，分为宏观和微观两步：

* **宏观（选地盘）：** 长音频的 VAD 阶段。主要通过计算声音强度的底噪，估算出一个动态阈值，把大块有声音的地方圈出来。决定“这个时间段对应哪个词”。
* **微观（找核心）：** 对每个词项寻找“元音核”。声调主要承载在元音和响辅音上，起头的爆破音（比如 p、t、k）通常是没有音高的杂音。软件会综合强度峰值、相对强度下降比例来锁定最值得采样的区域。

### 5.2 核心参数详解

调节以下参数，直接影响最终曲线的长相：

* **采样点数 (`pts`)：** 默认 **11**，范围 5 到 20。这表示在每个音节内部均匀采多少个 F0 点。传统的声调研究采 11 个点足以描绘出走势；如果你想做极其精细的微观变化研究，可以调大。
* **能量下降阈值 (`db`)：** 默认 **60**。它就像是切分微观元音核的一把尺子。数值越大，代表你越宽容，越容易把边缘比较微弱的声音（甚至是带点杂音的辅音）也框进提取范围；数值越小，条件越苛刻，边界切得越紧、越窄。如果你发现软件切得太窄，可以尝试增大它。
* **排除声母 (`skip_front`)：** 默认 **0**（秒）。这个参数的逻辑很精妙，它并不是简单粗暴地把声音开头固定切掉一截，而是给“元音起点检测算法”设置一个搜索的盲区。如果你的材料爆破音特别重，适当把这个值调成 0.05 或 0.1，可以有效避免软件把开头的塞音算作有效音高。
* **Pitch Floor (最低基频)：** 默认 **75 Hz**。告诉软件检测音高的下限。如果是嗓音极其低沉的男低音，可以把它调到 60 甚至 50；但如果是儿童或高亢的女声，不要设太低，否则软件很容易把环境嗡嗡声误认成音高。
* **Pitch Ceiling (最高基频)：** 默认 **600 Hz**。检测音高的上限。儿童和女声可以适当提高；普通成年男声如果不小心喊破音或者出现倍频错误（曲线突然翻倍飙高），可以把这个值降低（比如降到 350）来强行压制倍频误检。
* **清浊判定阈值 (`voicing_threshold`)：** 默认 **0.25**。这是判定声音到底是不是“浊音（声带震动产生音高）”的标准。数值越低，软件就越容易把那些能量很低、带有气声或者严重嗓音化（creaky voice）的片段也判定为有声；数值越高则越保守，宁可测出 0 也不瞎猜。
* **修剪边缘静音 (`trim_silence`)：** 默认开启。在词的边缘把低于 -50 dB 的极弱声音剔除掉。
* **F0 引擎 (`f0_engine`)：** 核心测音高算法。
  * **Praat：** 默认选择。使用的是经典的自相关 Pitch 方法，是学界公认的最稳妥选择。
  * **REAPER：** Google 参与开发的一种算法方案。它会先把音频重采样到 16kHz 再做平滑处理。在某些带有严重底噪或特殊发声态的材料上，它的表现可能会优于 Praat，也可能给出完全不同的结果。**强烈建议在同一个实验中从头到尾只使用同一种引擎。**

## 6. 数据导出与实验记录

辛苦修好的数据，必须能够无缝对接到你后续的统计软件中。PhonTracer 提供了极其丰富的导出选项。

### 6.1 导出格式全解析

* **TXT（纯文本）：** 最基础的导出格式。适合懂编程的用户，拿去用 Python 或 R 语言的脚本快速读取，里面包含了词项、分组和最原始的采样点数据。
* **XLSX（Excel 表格）[强推]：** 这是绝大多数用户的最佳选择。图形界面导出的 Excel 里不仅有原始的基频数据点阵列，软件还非常贴心地为你生成了一个 **`分析结果`** 工作表。在这个表里，不仅计算了最大值、最小值、均值，还直接用对数 Hz 归一化公式算好了 T 值！
  
  > **T 值的计算公式：**
  > `T = 5 * (log(Hz) - log(minHz)) / (log(maxHz) - log(minHz))`
* **TextGrid：** 把结果导回 Praat 中进行复查。如果是长音频，导出一个大的 `.TextGrid` 文件；如果是批量音频，则会导出一个包含多个小 TextGrid 的文件夹。
* **折线图 (Line Chart)：** 直接生成图片（支持 PNG、SVG、PDF）。它会把音节和采样点展开到横轴上，纵轴使用归一化的 T 值，可以非常直观地看出声调的轮廓走势。
* **KDE 热力图 (KDE Heatmap)：** 核密度估计图。这个图非常高级，它保留了那些被橡皮擦抹掉的 F0 空洞，用相对时间作横轴、T 值作纵轴，并通过颜色的深浅（热力分布）来展示声调轨迹最集中的区域。强烈建议勾选默认的“5%-95% 分位范围”来生成，这样可以最大程度屏蔽掉极端的误检点对图形范围的干扰。

### 6.2 导出编号规则与多发音人选项

导出数据时，软件会问你想要什么样的编号：
| 编号规则 | 含义 |
| :--- | :--- |
| **连续编号 (Continuous)** | 整个项目里所有的词项，不管在哪个分组，从 1 一直编号到最后（如 1, 2, 3... 100）。 |
| **组内编号 (Per Group)** | 重新编号！每个分组里的词项都从 1 开始算（如 A组 1,2,3；B组 1,2,3）。如果你需要严格对应原始材料里的实验编号，请谨慎选择。 |

如果你的项目里包含了多个发音人，在导出时会弹出额外的选项：

* **只导出当前发音人。**
* **分别导出全部发音人：** 为每个人生成一份独立的文件。
* **整合导出全部发音人：** 极其强大的功能！它会把所有发音人的数据糅合在一张大表或一张折线图中。不过请注意，TextGrid 格式不支持整合导出。

### 6.3 项目文件 (.teproj) 与自动保存

请千万记住：**仅仅导出 Excel 和图片是不够的！** 每天实验结束前，一定要点击 **导出项目 (.teproj)**。
`.teproj` 本质上是一个特制的压缩包，它不仅保存了你的所有边界修改和参数，甚至还把**音频的副本**和**F0 缓存数据**都打包进去了。这意味着哪怕你以后把原始录音弄丢了，只要有这个工程文件，随时随地都能在任何一台电脑上无损恢复工作状态。

虽然软件内有自动保存功能（开启后大约每 30 秒会在系统的 `.phon_tracer` 隐藏目录下保存一次备份），但这只是为了防止电脑死机造成的意外。正式交付数据前，请依然手动导出项目文件。

## 7. 进阶利器：AudioToolkit 独立处理套件

有时我们收到的录音极其混乱：要么是一段长达一小时却没有字表的乱弹，要么是几百个命名乱七八糟的微信语音。此时，建议你在进入 PhonTracer 主体前，先开启 `AudioToolkit` 独立音频处理套件给材料“洗个澡”。

### 7.1 合并音频（碎片缝合）

如果你想用长音频模式的连贯工作流，但手头全是一堆碎音频，用它！

1. 把多个 `.wav` 或 `.mp3` 拖进去。
2. 你可以随意拖动它们调整排队顺序。
3. 设定文件间的“静音间隔”（默认 0.5 秒，这能帮助 VAD 算法极好地认出边界）。
4. 点击合并。软件不仅会把它们无缝拼接，还会自动统一重采样到标准的 44100 Hz，输出一个纯净无暇的长 WAV 文件。

如果你有一份字表，这里还有一个绝活：**“导入字表自动排序”**。只要音频文件名里包含了字表里的词，软件就会自动把文件名跟字表做模糊匹配，瞬间把文件按字表顺序排得整整齐齐，没匹配上的会自动扔到最后面。

### 7.2 拆分长音频（一键切块）

假设你在做语料库整理，要把一长段录音切成几百个带标号的小文件。

1. 载入你的长录音文件。
2. 粘贴或导入字表（一行一个词）。
3. 点击“一键匹配”，软件会自动进行宏观的语音活动检测。如果发现切得不对，可以进入分段编辑器微调。
4. 设定边缘缓冲时间（比如默认留 0.1 秒的白头白尾，听起来更自然）。
5. 点击拆分，你会瞬间得到一个目录，里面的文件被完美命名为 `001_北京.wav`、`002_上海.wav`... 文件名里如果存在系统不允许的非法字符，软件也会聪明地自动替换掉，绝不报错卡死。

## 8. 批处理与 AI 助手：CLI 命令行完全指南

随着大语言模型（LLM）的爆发，越来越多人习惯用类似 Trae、Cursor、Codex 这样的 AI 编程助手来干活。为了满足极客用户和 AI 自动化的需求，PhonTracer 提供了强大的纯文本命令行界面（CLI）。

它能干几乎所有图形界面能干的事，并且所有的输出都是结构化的 JSON 格式，AI 工具读起来简直如鱼得水。

### 8.1 启动方式与基本语法

如果你是源码运行，在终端输入 `python cli.py` 即可进入交互模式；Windows 打包版则是双击运行 `PhonTracerCLI.exe`。
你可以随时输入 `help` 查看所有可用指令，或者输入 `status` 查看当前发音人和参数状态。

### 8.2 核心指令词典

这里列出在命令行中最常用的指令，它们完全可以用空格隔开参数：

* **全局设置类：**
  
  * `lang zh` / `lang en`：切换输出语言为中文/英文。
  * `speakers`：列出当前所有的发音人。
  * `add_speaker 发音人2`：新建一个名为“发音人2”的工作区。
  * `set_params pts=11 db=60 pitch_floor=75 f0_engine=praat`：这句命令直接把采样点、能量阈值、基频下限和算法引擎全改了。你也可以只改其中一两个。

* **导入类指令：**
  
  * `load_long "我的录音/长录音.wav"`：导入长音频。
  * `load_batch "录音文件/001.wav" "录音文件/002.wav"`：导入多个小音频。
  * `apply_wordlist "我的字表/字表.txt" fuzzy`：应用字表。后面的 `fuzzy` 代表模糊匹配；如果你想按顺序生套，可以写成 `order`。

* **检查与修改类指令：**
  
  * `list_items warnings`：让系统把那些提取出 0 或者报错的词全部列出来（相当于界面上的“需检查”筛选）。
  * `modify_bounds 3 1.250 1.820`：极度精确的微调！把编号为 3 的那个词，起止时间强行改为 1.250 秒到 1.820 秒。
  * `recalculate`：根据当前设置的参数全局重新计算。注意：这会覆盖掉你之前手工调过的边界，请谨慎使用！

* **导出类指令（格式：`export <格式> <输出路径> [编号规则] [目标]`）：**
  
  * `export xlsx "输出/结果.xlsx" continuous active`：把当前发音人（active）的数据以连续编号（continuous）的方式导出为表格。
  * `export line_chart "输出/图表.png" per_group integrated`：把所有发音人的数据整合（integrated）到一起，按分组编号，画出一张折线图保存下来。

### 8.3 管道与批处理文件的终极魔法

CLI 最强大的地方在于它可以被自动化脚本“操控”。你可以把一大串命令写在一个文本文件（比如 `commands.txt`）里，或者通过 PowerShell 的管道一次性喂给它。

**【示例 4】长音频自动提取的 PowerShell 批处理脚本：**

```powershell
@'
lang zh
load_long "我的录音/多语言.wav"
apply_wordlist "我的字表/多语言.txt"
export txt "自动化测试/long_results.txt" active
export xlsx "自动化测试/long_results.xlsx" per_group active
export textgrid "自动化测试/long_results.TextGrid"
project_export "自动化测试/保存的项目.teproj"
exit
'@ | python cli.py
```

**【示例 5】批量音频自动提取的 PowerShell 批处理脚本：**

```powershell
@'
lang zh
load_batch "散音频/001_北京.wav" "散音频/002_上海.wav" "散音频/003_广州.wav"
apply_wordlist "我的字表/字表.txt" fuzzy
export xlsx "自动化测试/batch_results.xlsx" continuous active
exit
'@ | python cli.py
```

### 8.4 给 AI 助手的专属提示词（Prompt）

如果你想让 Trae 或者 Codex 帮你代劳，不要让它们去直接修改源代码或者底层文件，这很容易搞坏系统。请直接把下面这段话喂给 AI：

> **“请在当前项目目录中使用 PhonTracer CLI 完成处理，绝对不要直接修改源码或手动篡改 .teproj 内部缓存文件。**
> **你的执行步骤如下：**
> **1. 运行 `python cli.py` 开启控制台。**
> **2. 使用 `load_long` 或 `load_batch` 指令导入我指定的音频。**
> **3. 使用 `apply_wordlist` 应用标注。**
> **4. 必须使用 `status` 和 `list_items warnings` 来检查结果。**
> **5. 如果有警告，尝试用 `modify_bounds` 修正。**
> **6. 导出 xlsx。**
> **7. 只要 CLI 返回的 JSON 里出现 `success: false`，立刻停止后续导出，先根据 `message` 字段排查路径和参数的错误，最后把你完成的操作总结给我。”**

通过这段提示词，AI 就能化身为你最得力的科研牛马，安安分分地走正规渠道帮你处理成千上万条数据了。

## 9. 常见疑难杂症与排雷指南

使用过程中难免遇到水土不服，我们总结了最常被问到的“翻车”场景及其急救方案。

**Q1：自动切分出来的片段数量，怎么跟我的字表对不上？**
**症状：** 软件提示某些词“缺失”，或者后面的词全错位了。
**急救：** 最大的可能是长录音里的词连读了没停顿，或者是中间发音人咳嗽了一声被算成了一个词。打开图形界面中间的“可视化切分器”，找到那个多出来的噪音段点右键删掉，或者在没识别出来的波形上点右键强行加一段，排好序后一切就会迎刃而解。

**Q2：提取出来的 F0 曲线全是断点，数据里一堆 0 怎么办？**
**症状：** 图表里坑坑洼洼，均值没法看。
**急救：** 首先看声音底噪是不是太大，或者这段其实是个清辅音（根本没音高）。如果是正常的浊音但依然测不出来，去左侧把 **Pitch Floor** 降低，把 **Voicing Threshold**（清浊判定阈值）调低一点让算法宽容些。如果还不行，果断切到 REAPER 引擎试试。

**Q3：音高曲线原本平滑，突然有一段冲上了天（或者掉进了地）？**
**症状：** 典型的倍频/半频错误。
**急救：** 算法把音高算成了一倍或一半。男声容易错高，女声容易错低。如果只错了一两个点，别去调参数了，直接点开“橡皮擦”工具，把那个飞上天的点一擦抹平，是最快的处理办法。如果是大面积错位，尝试降低或升高 Pitch Ceiling。

**Q4：批量匹配时，总是对应错文件？**
**急救：** 检查你的文件名。模糊匹配是很聪明的，但如果你的文件名全是类似 `001-a.wav`, `002-b.wav`，而字表里写的又是完全不同的东西，它肯定会瞎匹配。如果你能保证文件在文件夹里的排序跟字表完全一致，直接用 `order`（顺序匹配）是最高效的防错手段。

**Q5：导出的 TextGrid 怎么长得跟我预想的不一样？**
**急救：** 请注意你当前的模式。批量模式下，软件不会导出一个大 TextGrid，而是会导出一个文件夹，里面每个音频对应一个小 TextGrid。另外，如果是多发音人模式，软件是不支持“整合导出全部发音人的 TextGrid”的，必须分别导出。
