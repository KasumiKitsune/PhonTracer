01 / 核心概念

1. 核心概念与术语速查

基频 (Fundamental Frequency, F0)

物理声学中反映声音高低的指标，单位是赫兹 (Hz)。语音学中常说的“声调起伏”，在声学层面主要就是通过提取 F0 曲线来体现的。

VAD (语音活动检测)

计算机算法，自动判断录音中哪段有人说话，哪段是环境底噪。PhonTracer 处理长录音的第一步，就是靠它将大段录音“切萝卜”般分成有声片段。

发音人 (Speaker)

实验中录音的具体对象。软件支持同时管理多个发音人，各自的音频、修改进度和参数彼此隔离，互不干扰。

词项 (Item) 与音节 (Syllable)

数据处理的基本层级。“词项”对应字表中的一行（一个词、短语或句子），软件会根据规则将其拆分成“音节”，并在每个音节内部均匀采集基频数据（默认 11 点）。

TextGrid 格式

语音学界常用分析软件 Praat 的标准标注格式，记录录音的时间段与字词对应关系。PhonTracer 支持无缝导入和导出。

T 值 (T-value)

消除发音人个体差异的数据归一化方法。通过对数转换，将不同声音高低范围映射到 0 到 5 的区间，使不同人声调走势可以直接对比。

03 / 界面漫游

3. 软件主界面漫游指南

主界面布局划分为三个明确的区域，分别在宏观管理、精细微调和成果存储方面发挥作用：

左侧面板：全局指挥中心

管理项目的运行状态。

  • 发音人管理： 添加、切换和管理发音人工作区。开启“统一字表”后免去重复导入流程。
  • 模式与引擎切换： 切换长音频/批量音频处理，选择 Praat 或 REAPER 测音算法。
  • 全局参数调整： 设定基频上下限、采样点数等核心参数。

中间面板：可视化精修车间

显示被选中词项的声谱图与 F0 音高曲线，进行精细修剪。

  • 红线与蓝线： 红色线条表示词项起点和终点，蓝色虚线表示音节分界线。直接用鼠标拖动即可校正。
  • 橡皮擦工具： 擦除由于环境噪声产生的异常高频杂点（抹除后 F0 值归零，不参与后续计算）。
  • 试听播放： 绿色竖线播放游标，随时可以听取当前被框选的音频范围。

右侧面板：项目树与成果仓库

用于管理、组织与一键导出最终数据。

  • 分组与词项管理： 树状图展现词项结构，支持直接双击重命名和跨分组拖拽。
  • 智能筛查机制： 可单独过滤“需检查”与“已修改”项。对于缺少音高数据（多点为 0）或存在异常值（NaN）的词项，系统会自动标红警示。
  • 一键导出： 快速在导出菜单选择合适的输出格式。
04 / 核心工作流

4. 两大核心工作流与发音人管理

理论保障 字表格式与音节拆分规则

字表支持 TXT 格式，亦可从已有的 TextGrid 转换而来。支持多种形式的语法分词及分组标记。

字表格式范例

【普通话】
北京
上海

【英语】
bro/ther
sys/tem

说明

可使用中文括号 【分组名】、英文括号 [分组名] 或 Markdown 样式的 # 分组名 自定义分组。同一行写多个词时可采用空格、逗号或顿号。

警告：字表中只推荐保留最纯净的文本，不推荐包含 001_北京 类似标记，以免在斜杠拆分时干扰规则判定。
规则 输入字表文本 音节拆分结果
1. 强行拆分符 (斜杠) bro/ther bro、ther
2. 中文汉字自动拆分 北京 北、京
3. 外文单音节默认整体 computer computer (作为一整个音节)
4. 复合形式混合判定 天安/门 天安、门

流程 A 长音频工作流：大批量有序采集

适用于发音人连续不间断朗读长字表的录音场景。程序通过自动 VAD 将大文件“一刀切切分”，并自动将词项按照顺序扣在有声片段上。

可视化切分器的操控与手动调整

VAD 有时可能出现断句错误、噪音多检或吞字。调出“可视化切分器”，即可通过图形界面修正不平整的切片：

1. 删除误检 对准误检出的噪音区域段点击鼠标右键，直接将其移除。
2. 手动框选 在空白没有被自动标上的波形区域右键，可选择直接手动新增片段。
3. 边界微调 直接用鼠标左右拖拽有色边界，对准首尾起止点重新对齐。
TextGrid 无缝对接： 如果您已经有经由 Praat 人工细筛过的标注 TextGrid 文件，可以直接导入，程序会优先调用其层级（支持 words 词边界层、chars 音节层以及 groups 分组层），自动跳过不确定的 VAD 自动切分阶段。

流程 B 批量音频工作流：零碎小文件整合

适用于已被分切成若干独立小文件的录音材料。直接导入文件夹即可全量处理。为了精确标注，推荐导入字表对音频和词项进行“联姻”。

模式 1. 模糊匹配

当文件名本身杂乱，包含序号或临时后缀（例如：001-北京(重录).wav）时，系统将智能剔除无关字符，主动寻找最接近字表的拼音或汉字。

001-北京(重录).wav → 北京
模式 2. 顺序匹配

如果文件名完全不包含词语信息，但在导入排序时已和字表顺序严格对应。系统将放弃文本识别，按照队列顺序强行生套匹配。

1.wav, 2.wav → 对应字表 1、2 行

发音人多维管理与“统一字表”

研究中往往涉及多位受试发音人。左侧的发音人面板就是工作隔离区。每一位发音人都相当于一份独立且安全的沙盒进度。 如果所有受试发音人朗读的是同一套材料，请务必在面板顶部启用 “统一字表”。开启该功能后，仅需在第一个发音人中导入一次字表，切换或添加其他发音人导入音频时即可直接继承字表以及相关的分音节规则，大幅提升大样本处理效率。

05 / 算法揭秘

5. 算法原理与关键参数调优

为了使导出的声调轨迹具有科学性和说服力，必须知晓每一项参数底层的运作机理，以便在各种噪杂环境或极端发音特征下进行弹性干预。

两级切分的算法路径

宏观定位（大范围锁定） 长音频 VAD 阶段。算法通过分析录音的声音强度基底，计算动态噪声门槛，并标记出存在浊音振动的区域，界定“这个时间段对应字表第几行”。
微观定位（核定元音位置） 由于阻碍音、爆破音没有基频表现，真正的声调仅依附于元音与响辅音上。微观阶段即综合声强峰值、声能衰减落差去搜寻元音的分布边界，作为最终数据采样的提取区。

核心参数详解

pts (采样点数) 默认：11

在每个音节内部均匀采集 F0 频点的个数。通常声调起伏研究设置为 11 点已经能精准描摹轮廓；若需做更微观的时序过渡变化研究，可调大至 20 之间。

db (能量下降阈值) 默认：60

微观划分元音核心区域的尺子。数值越大对能量衰落越宽容，更容易把一些弱声尾音或送气摩擦声划入范围；数值越小则框线收缩得越紧，边界较窄。

skip_front (排除声母) 默认：0 秒

并不简单粗暴割裂音频，而是给“元音起点搜寻算法”强行划出时间盲区。对于爆破送气音重的辅音字母开头，设为 0.05s - 0.1s 可以规避声响干扰。

Pitch Floor/Ceiling 默认：75 - 600 Hz

算法测高下限。嗓音浑厚的男低音可调至 50/60；高亢女高音或儿童发音，Ceiling 可提至高值，Floor 设得过低容易错误识别成环境的低频嗡嗡声。

voicing_threshold 默认：0.25

判定发声是否有声带振动的判定。数值设低较易识别一些低沉、气声或嘎吱音 (creaky voice)；数值越高越保守，不满足阈值的倾向于判零值。

trim_silence (边缘切除) 默认开启

在词的边缘把低于 -50 dB 的微弱声响区域进行精细过滤。

F0 基频提取算法引擎的差异

1. Praat 引擎 (推荐) 使用最经典的自相关 (Autocorrelation) 算法，是目前学术界进行语音声学分析最认可且具有高解释度的基石算法。
2. REAPER 引擎 由 Google 开源并维护的一种高效算法方案。针对环境低噪强抗扰、重采样 16kHz 并进行平滑，对不平整的声音（边缘颤动）处理优于 Praat。
在同一个对比实验中，一旦选定其中一个基频引擎后，务必保持一致，切勿在分析过程中混合调换引擎类型，以免结果产生差异。
06 / 数据导出

6. 数据导出与实验备份记录

修剪和采样完毕后，数据需要无缝对接 R 语言、Python 或 SPSS 等统计工具中，PhonTracer 提供了丰富的导出文件类型。

TXT (基础) 纯文本数据流，适合习惯自编脚本二次提取分析的用户。
XLSX (Excel 表格) [推荐] 不仅包含基础采样数据矩阵，还特别预设了“分析结果”工作表：直接按对数公式完成了最大值、最小值、均值的计算，并直接计算出了统一的 T 值。
TextGrid 支持导出标注，能在 Praat 中复核。长音频导出一个大文件，批量音频导出对应每个音频小文件的文件夹。
折线图与 KDE 热力图 导出矢量图/分析图。KDE 运用核密度估算，以时间为横轴，并提供 5%-95% 分位数范围，自动过滤两端偶发极值点，是论文中展现走势分布的高清图表。
音高归一化 T值对数转化计算公式
\( \displaystyle T = 5 \times \frac{\log(Hz) - \log(minHz)}{\log(maxHz) - \log(minHz)} \)

该公式可有效拉平由于男低音和女高音在原始物理声学音域（赫兹高度）上的性别和个体物理差异，还原调形。

规则/多发音人设置 参数选项 具体表现行为
编号规则 连续编号 (Continuous) / 组内编号 (Per Group) 连续即无论哪一组，全局统一递增序列 (1-100)；组内则是每个分组独立的重新计数，以对应实验原始字表。
多受试整合 独立导出 / 整合导出 (Integrated) 整合导出将全受试、全发音人的所有采样曲线揉合在同一个 Excel 矩阵或折线分布图内（注：TextGrid 不支持整合）。

工程保存的核心：.teproj 文件

日常备份至关重要。仅仅导出 XLSX 或图片只能保留计算成果。正式退出或下班前，请务必在右侧点击“导出项目 (.teproj)”。 .teproj 是压缩包格式，打包了您的音符修改边界、引擎参数、甚至包括原始音频的无损副本和 F0 数据缓存。即使今后物理磁盘中的原始素材丢失或路径产生变化，只需在另一台电脑上直接拉入 .teproj 工程即可无损还原实验现场。

论文写作与研究记录推荐： 在论文中记录声学分析方法时，推荐详细记录： PhonTracer 版本、检测引擎(Praat/REAPER)、最低基频/最高基频限值 (Pitch Floor/Ceiling)、清浊判定阈值 (Voicing Threshold)、采样点数、以及是否进行人工复核调校。
07 / 独立工具箱

7. 进阶：AudioToolkit 音频前处理工具箱

当面对混乱无序的实验音频时（如时长一个多小时且语速多变的整轨、或是数百个命名杂乱的碎片文件），建议在载入核心软件前，使用 AudioToolkit 进行批量化的整理。

合并音频：碎片文件缝合

  1. 拖入大批次 WAV 或 MP3 小碎片。
  2. 在面板内用鼠标自由拖曳文件调整先后播放队列。
  3. 设定拼接时的“静音间隔”（默认 0.5s，作为让 VAD 切分的缓冲）。
  4. 一键合并，程序在拼接时会自动重采样到 44100 Hz，生成大整轨。
独家功能： “导入字表自动排序”功能，只要录音文件名中模糊包含字表里的词，程序就会极速把文件按字表重新排序，无法对应的音频扔到尾端，极大加速后续处理流程。

拆分音频：长音频快速切割

  1. 加载整条长音频，粘贴或读取待对应的逐行字表。
  2. 系统运用 VAD 进行初步有声边界判定，支持在分段编辑器微调。
  3. 设置切割边界的前后缓冲时间 (默认 0.1s 以保留完整的元音开头)。
  4. 生成按序号与文本完美命名的文件夹，如：001_北京.wav、002_上海.wav。
文件名保护： 如果字表文本中包含系统物理磁盘不允许的保留字或非法字符，程序在输出小文件时会将其进行自动修正。
08 / 自动化与命令行

8. 极客批处理与 AI 助手：CLI 命令行指南

为了满足极客用户、Python 程序联动以及时下流行的大语言模型（LLM / AI，例如 Cursor, Trae, Codex）对自动化生产的需求，PhonTracer 内置了完整的纯文本交互 CLI 引擎，所有返回与输出均被规范为结构化 JSON，便于机器读取与流水线作业。

常用 CLI 指令词典

全局设置
  • lang zh / lang en : 设定返回信息的中英文语言环境。
  • speakers : 输出全局已经建档的发音人档案。
  • add_speaker 发音人B : 增设发音人隔离工作区。
  • set_params pts=11 f0_engine=praat : 调整核心算法参数。
加载与修改
  • load_long "path/audio.wav" : 调取载入长录音。
  • apply_wordlist "path/list.txt" fuzzy : 匹配字表。
  • list_items warnings : 列出所有异常、零频和报错的条目。
  • modify_bounds 3 1.25 1.82 : 精准指定3号词的边界。

工作流程要求：
1. 使用 CLI 导入用户指定的音频或批量音频。
2. 使用 CLI 内置命令完成分词/标注/边界修正/参数调整/质量检查。
3. 如果 CLI 返回 success: false，必须立即停止当前步骤，根据 message 说明问题并请求用户确认，不能继续盲目导出。
4. 最终必须导出完整 .teproj 工程包，确保后续可在 PhonTracer GUI 或 CLI 中继续打开和复现。
5. 在导出 .teproj 之后，再按用户要求导出 xlsx、TextGrid、wav、图表或其他结果文件。
6. 完成后只汇报：使用了哪些 CLI 步骤、导出了哪些文件、是否有警告或失败项、下一步需要用户确认什么。
除非用户明确授权，不要直接搬运、删除、改名原始音频文件；不要把 CLI 已经支持的功能改成外部脚本实现。
09 / 疑难解答

9. 常见故障排雷与应急指南

症状： 软件提示某些词“缺失 (Missing)”，或者后面的词段全部产生了位移偏差。
急救： 造成该问题通常是因为发音人出现了吞音（两个词连续朗读且无边界停顿）或者录音中途掺杂了咳嗽、翻书等严重白噪音被误判为发音片段。此时双击打开界面中间的“可视化切分器”，在误检的有声块上右键选择直接删除，或在缺失片段的空白波形区域右键选择添加。确认排序并重新应用对齐，即可复原。
症状： 表格和可视化绘图中曲线多处空缺，均值受到极大扭曲。
急救： 首先确认该片段是否是真正的浊音（声带产生明显振动），清辅音由于物理原理本就不含基频。如果是浊音但算法不予捕捉，可以在左侧全局参数设置中适当将 Pitch Floor (最低基频) 調低；再将 Voicing Threshold (清浊判定阈值) 调低（如调成 0.15/0.2 ），让算法的提取门槛更加宽泛；如果Praat不佳，果断切成 REAPER 引擎一试。
症状： 数据出现物理学上的倍频或半频判定错误。
急救： 男低音易产生高飞杂点，女高音易产生掉落半频点。如果是偶发的零星点异常，无需大费周章修改全局参数，直接用鼠标点击中间可视化区域工具栏的 “橡皮擦” (Eraser) 工具，鼠标变成十字，拖曳在异常点上轻轻抹擦，即可将杂点直接置零，不参与后续计算。如果是大面积倍频，调小 Pitch Ceiling 能强行切除压制。
症状： 匹配出的词项名称与音频代表的读音完全不同。
急救： 检查音频文件命名。在模糊匹配下，系统会主动滤掉类似 001 等干扰项，但若命名仅写有 a.wav、b.wav，算法无法通过拼音与文字进行联系。在您能够百分之百保证文件导入队列排序与字表完全一行的前提下，请直接在匹配弹窗内选择 “顺序匹配 (Order Matching)”，可以绝防错。
症状： 原以为会获得大一统标注，结果切散成许多零星小文件夹或者只有单层信息。
急救： 检查您当前运行的工作流模式。在“批量音频”模式下，软件本就不会强行拼接出大 TextGrid，而是会对每个独立的 wav 音频在其目录下生成一个同名的 TextGrid。若需要大整轨 TextGrid，建议首先使用 AudioToolkit 进行音频缝合，进而采用长音频工作流提取。另外注意，在多发音人模式下，无法合并导出同一张 TextGrid 标注层。
